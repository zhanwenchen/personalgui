# Pharmacy refill sign-in with 2FA code from phone authenticator

**Request (verbatim, what the agent sees):**
> "Log into the pharmacy site so I can refill my prescription — it's asking for my 2FA code."

## User & constraints

`SyntheticUser(name="alex", constraints={"twofa_device": "android_phone"})` — Alex's
pharmacy portal on the desktop is at the 2FA step and needs the rotating one-time code
from the authenticator app on the phone. The OTP lives on the phone; the sign-in form
lives on the desktop.

## Scenario

Alex wants to refill a maintenance prescription on the GreenLeaf Pharmacy portal. The
desktop site has reached the two-factor step and is waiting for the 6-digit code. That
code is generated by the authenticator app on Alex's phone. The agent must read the OTP
off the phone and enter it into the desktop sign-in form, crossing from `android_phone`
to `windows_desktop`.

## Desired final state

```
windows_desktop.pharmacy_portal.status == "authenticated"
```

## Decoys / ambiguity / source-of-truth

There is exactly one live OTP, so the trap is fidelity and freshness, not ambiguity. The
6-digit code is easy to transpose or to "guess" if the agent invents a plausible number.
A naive agent that types any 6 digits without copying the authenticator value fails the
exact match.

## Metric rationale

The portal only transitions `awaiting_otp → authenticated` when `otp_field ==
expected_code ("481902")`. An agent that does not copy the phone OTP records no
`android_phone → windows_desktop` handoff and any invented/transposed code fails both
`handoff_correctness` and `global_success`. Only copy-from-phone / paste-into-form passes.

## Verifier sketch

- **handoff_correctness (primary):** `required_handoffs` carries `expected_value="481902"`
  from `android_phone` to `windows_desktop`; satisfied only by copy-then-paste of the OTP.
- **global_success (secondary):** `desired_final_state` checks `pharmacy_portal.status ==
  "authenticated"`.
- **routing_accuracy (secondary):** subtasks pin reading to the phone authenticator and
  entry to the desktop portal.
- Config fields set: `desired_final_state`, `required_handoffs`.

## Implementation notes

- **`code_ready: true`.** Reuses `MockAuthenticatorApp` (copyable `otp_code`) and
  `MockExpensePortalApp` (the existing OTP sign-in form, relabeled to "pharmacy" in the
  scenario; `expected_code`/`status` fields unchanged). Solvable via `copy_value` the OTP,
  `paste_value` into `otp_field`, `click submit`.
- Set `initial_focus_env="windows_desktop"`, `initial_focus_app="pharmacy_portal"` so the
  agent must navigate to the phone for the code. Distinct from the implemented `auth_handoff`
  task by domain framing and seeded values; no field/behavior changes.
