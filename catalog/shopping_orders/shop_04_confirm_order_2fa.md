# Confirm a high-value order with the phone authenticator code

**Request (verbatim, what the agent sees):**
> "Confirm my Halverson camera order — it needs the 2FA code from my authenticator."

## User & constraints

`SyntheticUser(name="alex", constraints={"otp_source": "android_phone"})` — the camera is a
high-value purchase, so the retailer requires a one-time 2FA code to confirm the order. The
code is generated by the authenticator app on Alex's phone; the order-confirmation web form
is on the desktop. This forces a phone -> desktop handoff.

## Scenario

Halverson Photo's checkout has reached the confirmation step for an expensive order and is
prompting for a 2FA code. The current code is shown only in the authenticator on the
`android_phone`. The agent must read the OTP on the phone and enter it into the
order-confirmation web form on the `windows_desktop`, then submit to finalize the order.

(Cross-env handoff: `from_env=android_phone -> to_env=windows_desktop`.)

## Desired final state

```
windows_desktop.confirm_form.status == "submitted"
```

## Decoys / ambiguity / source-of-truth

The trap is fidelity across devices: the OTP is six digits and must cross from the phone to
the desktop exactly. A naive agent that hand-types a remembered/misread code, or that tries to
"confirm" without fetching the phone code at all, never records the cross-env handoff and
fails the form's `expected_fields` match. There is exactly one order and one code, so the
difficulty is the device-spanning carry, not referent ambiguity.

## Metric rationale

An agent that completes the literal request on the desktop alone (no phone read) never produces
the `704913` value, so no `android_phone -> windows_desktop` handoff is recorded and the form
stays in `drafting`. Only copy-from-authenticator / paste-into-form both records the handoff and
matches the code.

## Verifier sketch

- **handoff_correctness (primary):** `required_handoffs` carries `expected_value="704913"` with
  `from_env=android_phone`, `to_env=windows_desktop`; satisfied only when the phone OTP is copied
  and pasted into the form input.
- **global_success (secondary):** `desired_final_state` checks `confirm_form.status ==
  "submitted"`, set when `fields["otp"] == expected_fields["otp"]`.
- **routing_accuracy (secondary):** the OTP read pinned to `android_phone`/`authenticator`, the
  submit pinned to `windows_desktop`/`confirm_form`.
- Config fields set: `desired_final_state`, `required_handoffs`.

## Implementation notes

- **`code_ready: true`.** `MockAuthenticatorApp` seeded `otp_code="704913"` (copyable);
  `MockBrowserFormApp` seeded `expected_fields={"otp":"704913"}`, `status="drafting"`. Solvable
  via `copy_value` on the phone OTP, switch env to desktop, `paste_value` into `otp`, `click
  submit`. `desired_final_state` path references a real `MockBrowserFormApp` key.
- Set `initial_focus_env="windows_desktop"`, `initial_focus_app="confirm_form"` so the agent must
  navigate to the phone to fetch the code, mirroring the implemented `auth_handoff` mechanics.
