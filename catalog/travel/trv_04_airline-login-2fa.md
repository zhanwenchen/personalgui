# Airline account login needs a 2FA code from the phone

**Request (verbatim, what the agent sees):**
> "Log me into my Aerolux account on the desktop."

## User & constraints

`SyntheticUser(name="alex", constraints={"twofa_device": "android_phone"})` — Alex's
Aerolux account uses two-factor authentication. The desktop sign-in shows a one-time-code
prompt; the live OTP is generated by the authenticator app on the phone.

## Scenario

Alex is signing in to the Aerolux frequent-flyer portal on the desktop to manage a
booking. The portal has accepted the username/password and now shows an OTP challenge. The
current 2FA code is only visible in the phone authenticator. The agent must read the code
on the phone and enter it in the desktop sign-in form — a phone-to-desktop handoff.

| `windows_desktop` | `MockExpensePortalApp` | `otp_field=""`, `expected_code="704218"`, `status="awaiting_otp"`, page labelled "Aerolux sign-in" |

## Desired final state

```
windows_desktop.signin.status == "authenticated"
```

## Decoys / ambiguity / source-of-truth

The authenticator also shows an expired/previous code line ("prev 311902") and a separate
account tile for an unrelated service ("StreamPlus 558740"). Only the live Aerolux code
`704218` matches `expected_code`. A naive agent that types a remembered or wrong-account
code fails the portal match. There is one correct code, so the trap is selecting the
right tile and transferring it faithfully.

## Metric rationale

An agent that types a code from memory or grabs the wrong account's OTP never records a
`from_env=android_phone → to_env=windows_desktop` handoff with the correct value, so the
portal stays `awaiting_otp` and `global_success` fails. Only copy-from-phone /
paste-into-form satisfies both the handoff and the match.

## Verifier sketch

- **handoff_correctness (primary):** `required_handoffs` carries `expected_value="704218"`
  from `android_phone` to `windows_desktop`; satisfied only when the phone OTP is copied
  and pasted into the form.
- **global_success (secondary):** `desired_final_state` checks `signin.status ==
  "authenticated"`, set when `otp_field == expected_code` on submit.
- **routing_accuracy (secondary):** reading pinned to phone authenticator, entry to
  desktop sign-in.
- Config fields set: `desired_final_state`, `required_handoffs`.

## Implementation notes

- **`code_ready: true`.** `MockAuthenticatorApp` on `android_phone` with copyable
  `otp_code`; `MockExpensePortalApp` on `windows_desktop` (reused as the OTP sign-in form)
  seeded `expected_code="704218"`, `status="awaiting_otp"`. Solvable via `copy_value` on
  the phone code, `paste_value` into `otp_field`, `click submit`. Mirrors the existing
  `auth_handoff` pattern but framed as an airline account; distinct seed values keep it
  from duplicating that task. `desired_final_state` path is a real key.
- Set `initial_focus_env="windows_desktop"`, `initial_focus_app="signin"`.
